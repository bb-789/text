Web 渗透测试基础学习笔记

今天开始系统整理 Web 渗透测试相关的知识，记录一下核心的概念和基本流程。郑重提示：所有安全测试必须在获得明确授权的情况下进行！ 遵纪守法是安全从业者的第一底线。

一、 渗透测试基本流程

规范的渗透测试通常遵循以下几个核心阶段：

  1. 前期交互阶段 (Pre-engagement)
    • 明确目标范围、测试时间、授权许可（必须拿到授权书）。
  2. 信息收集阶段 (Information Gathering)
    • 这一步决定了渗透的深度。包含：域名信息（WHOIS）、子域名探测、端口扫描、旁站/C段、指纹识别（CMS类型、开发语言、操作系统等）。
    • 常用工具：Nmap, FOFA, Wappalyzer
  3. 漏洞分析阶段 (Vulnerability Analysis)
    • 结合自动化扫描器（如 AWVS, Nessus）与手工抓包测试。
  4. 渗透攻击阶段 (Exploitation)
    • 尝试验证漏洞是否真实存在，获取初步权限。
  5. 后渗透阶段 (Post Exploitation)
    • 权限提升、维持访问、内网横向移动（需根据项目合同需求决定是否进行）。
  6. 报告阶段 (Reporting)
    • 整理漏洞清单，评估风险等级，并给出开发层面的修复建议。

二、 常见 Web 漏洞概览 (OWASP Top 10)

在日常测试中，最常碰到的几类基础 Web 漏洞如下：

1. SQL 注入 (SQL Injection)

  • 核心原理：后端程序未对用户输入的数据进行严格过滤或转义，导致恶意的 SQL 语句被拼接到数据库中当作代码执行。
  • 常见类型：联合查询注入、报错注入、盲注（布尔盲注、时间盲注）。
  • 防御方案：全面使用预编译（参数化查询），严格校验输入的数据类型。

2. 跨站脚本攻击 (XSS)

  • 核心原理：攻击者在网页中插入了恶意的 JavaScript 代码，当普通用户浏览该页面时，恶意代码在用户的浏览器中触发执行。常被用于窃取 Cookie 或钓鱼。
  • 常见分类：反射型 XSS、存储型 XSS、DOM 型 XSS。
  • 防御方案：对用户输入进行严格的 HTML 实体编码输出，敏感 Cookie 设置 HttpOnly 属性，合理配置 CSP（内容安全策略）。

3. 文件上传漏洞 (File Upload)

  • 核心原理：服务器侧没有对用户上传的文件类型、内容进行严格的校验和重命名，导致攻击者成功上传了恶意脚本文件（如 WebShell），从而控制服务器。
  • 防御方案：采用严格的后缀名白名单机制，校验文件头（Magic Number），隐藏上传后的真实文件路径，且务必取消上传目录的脚本执行权限。
